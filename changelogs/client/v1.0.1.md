### Fixed
- New sessions and forked sessions now properly register as participants on the server. Previously, creating a new session did not trigger a session resume, so the client was never registered as a participant and control state was never initialized. This caused all server-broadcast messages (including chat responses) to be silently dropped — the user would see nothing in the chat until manually clicking the session again to resume it.
- Refreshing the page on a session URL (e.g. `/server/$serverId/workspace/session/$sessionId`) now properly loads the session contents, messages, usage, and control state. Previously, the WebSocket connected before the session list was fetched from the REST API, so the "resume on connect" logic had no session ID to resume — leaving the chat empty and without control until the user manually clicked the session in the sidebar.
- Fixed Electron auto-updater resolving to the wrong GitHub release. The updater was looking for `latest-mac.yml` in the `server/v1.0.0` release instead of `client/v1.0.0`, resulting in a 404 error. This happened because `electron-updater` uses GitHub's `/releases/latest` endpoint which returned the server release instead of the client release (both are non-prerelease in the same repo with the same version). The updater now explicitly resolves the correct `client/v*` tagged release via the GitHub API and uses a Generic provider to fetch update artifacts from the right URL.
