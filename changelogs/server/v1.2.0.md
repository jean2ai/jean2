### Fixed

- **N+1 query on every chat turn and message load** — `listMessagesWithParts()` and `listMessagesForSession()` ran 1 query per message (1+N pattern), now use a single `LEFT JOIN` query with in-memory grouping. A 500-message session went from 501 queries to 1.
- **Orphaned data from session/workspace deletion** — `PRAGMA foreign_keys = ON` was never set in production, so `ON DELETE CASCADE` constraints silently did nothing. Deleted sessions left their messages, parts, pending_asks, queued_messages, and pinned_messages in the database forever. FK enforcement is now enabled at DB initialization.
- **`deleteSession()` data leak** — only deleted attachments and the session row, leaving all child data orphaned. Now wraps the delete in a transaction and relies on FK cascade for child cleanup, plus removes FTS index entries.
- **`deleteSessionsByWorkspace()` not transactional** — attachment cleanup and session deletion were separate auto-commit statements that could leave inconsistent state on partial failure. Now wrapped in a single transaction with FTS cleanup.
- **Silent title generation failures** — `session.generate_title` now sends error responses to the client on failure instead of swallowing exceptions silently. Both the "no title generated" and exception paths notify the requesting client.
- **Unhandled throws in delete/rename** — `session.delete` and `session.rename` handlers had no try/catch. Unexpected errors (e.g., DB failures) propagated to the top-level catch and were reported to the client as `parse_error`, which was misleading. Both now have proper try/catch blocks that send operation-specific error codes (`delete_error`, `rename_error`).
- **`sessionId` on all session error responses** — fork, revert, compact, edit, delete, rename, and title generation handlers now include `sessionId` on error sends so the client can correlate errors to pending operations and clear loading states.

- **CLI commands hanging after completion on macOS**: finite commands now exit explicitly after completing while preserving their exit codes. Long-running `server`, `logs`, and `_client` commands remain active as intended. The daemon launcher also closes its parent-side log file descriptor after spawning the server.

### Added

- **`jean2 db` CLI command** — database maintenance with `stats` (check size and reclaimable space), `vacuum` (remove orphaned data + reclaim free space), and `cleanup` (remove orphaned data only) subcommands.
- **`vacuumDatabase()` function** — runs orphan cleanup, WAL checkpoint (`TRUNCATE`), then `VACUUM` to rebuild the database file. Supports dry-run mode for stats-only reporting.
- **Maintenance REST routes** — `GET /api/maintenance/db-stats`, `POST /api/maintenance/vacuum`, `POST /api/maintenance/cleanup` for programmatic access.
- **Centralized HTTP error handling** — new `HttpError` hierarchy (`BadRequestError`, `NotFoundError`, `ForbiddenError`, `ConflictError`, `UnauthorizedError`, `PayloadTooLargeError`, `UnprocessableEntityError`) in `utils/http-errors.ts`. Route handlers now `throw new NotFoundError(...)` instead of wrapping every handler in try/catch. The `app.onError()` handler maps error types to status codes automatically.

### Changed

- **`listMessagesForSession()` now delegates to `listMessagesWithParts()`** — both functions were functionally identical (fetching all messages with parts), now share the optimized single-query path.
- **Route error handling consolidated** — removed ~410 lines of duplicate try/catch boilerplate across all 8 route files. `ConfigurationError` and `PinnedMessageError` hierarchies now extend `HttpError`, so their status codes are handled centrally. `config.ts` alone shrank from 532 to 285 lines.
- **`regenerateSessionTitle` signature** — now accepts `ws: ServerWebSocket` as a parameter so it can send error responses directly to the requesting client. The auto-title path after chat messages also passes `ws`.
