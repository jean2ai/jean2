### Added

- **Gmail OAuth service provider** — new Gmail provider (`providers/gmail.ts`) that connects via Google OAuth 2.0 with PKCE. Registered as `kind: 'service'` (not `'llm'`), meaning it does not implement `createModel()`. Tools read the stored access token directly from `~/.jean2/providers/gmail.json`. Credentials configured via `JEAN2_GMAIL_CLIENT_ID` and `JEAN2_GMAIL_CLIENT_SECRET` env vars. Redirect URI is hardcoded to `http://localhost:1455/oauth/gmail/callback`.
- **Background token refresh for Gmail** — proactive timer that checks every 5 minutes and refreshes the Gmail access token before it expires (5-minute buffer). Starts automatically on server startup if Gmail was previously connected, and on successful OAuth completion. Eliminates the need for tools to handle token refresh logic or 401 retries. Uses an `isRefreshing` guard to prevent concurrent refresh races.
- **`GmailProviderConfig` SDK type** — extends `OAuthTokenConfig` with an optional `email` field (extracted from the Google `id_token` JWT during OAuth completion). Stored at `~/.jean2/providers/gmail.json`.
- **`kind` field on `ProviderDescriptor`** — new optional field (`'llm' | 'service'`) distinguishing LLM model providers from data/action service providers. Codex remains `'llm'` (implicit), Gmail is `'service'`.
- **`clientSecret` on `OAuthProviderConfig`** — optional field for OAuth providers that require a client secret in token exchange and refresh calls (Google, etc.). Included in both `exchangeCodeForTokens()` and `refreshTokens()` when present.
- **Multi-path localhost callback server** — the OAuth manager's localhost `Bun.serve()` listener now supports multiple callback paths on the same port. Gmail (`/oauth/gmail/callback`) and Codex (`/auth/callback`) coexist on port 1455 simultaneously. Uses a `Set<string>` of registered paths with reference-counted server lifecycle (`stopLocalServerForPath` replaces the old `stopLocalServers` that killed all servers at once).
- **Workspace path management on `ToolContext`** — two new methods on the `ToolContext` interface (SDK): `addWorkspacePath(path: string): Promise<boolean>` and `removeWorkspacePath(path: string): Promise<boolean>`. These let tools dynamically register/unregister paths on the current workspace's `additionalPaths` at runtime, so all subsequent tool calls in the session can access the new directory without per-call permission prompts. Returns `false` when no workspace is associated with the session. Implemented server-side via `createWorkspacePathManager()` in `executor.ts`, which reads the workspace, appends or removes the resolved path, and writes via `updateWorkspace()`.

### Changed

- **`createModel()` is now optional on `ConnectableProvider`** — service providers (Gmail) can omit it. `createModelForProvider()` throws a clear error (`"Provider does not support creating models (kind: 'service')"`) when called on a service provider, instead of crashing.

### Removed

- **`JEAN2_GMAIL_REDIRECT_URI` env preset** — removed from `env-presets.ts`. The redirect URI is hardcoded in `gmail.ts` and only needs to be registered in Google Cloud Console, not configured as an env var. Was showing an incorrect example value (`http://localhost:1455/auth/callback` — the Codex path) that would confuse users.

### Added

- **Scheduled tasks (cron)** — full scheduling system for recurring and one-shot automated tasks. Jobs are workspace-scoped and run as normal top-level sessions using the existing agent engine. Four schedule types: one-shot (at a timestamp), interval (every N minutes), daily (at a time), and weekly (on specific weekdays at a time). Jobs support repeat limits, optional session reuse with configurable history inclusion, and per-job preconfig/model selection. A 60-second tick loop with at-most-once semantics handles execution. Scheduled sessions are tagged with `metadata.scheduledJobId` and grouped under their job in the sidebar Scheduled section. Jobs are cleaned up automatically on workspace deletion. Startup recovery runs an immediate tick to catch jobs that became due while the server was offline.
- **Scheduler workspace capability** — a built-in `scheduler` tool (action-style: create/list/update/pause/resume/trigger/remove) that lets the agent manage scheduled tasks via natural language. Enable per-workspace in Settings > Scheduling with a configurable permission risk level. The schedule tool is stripped from spawned sessions to prevent recursive scheduling loops.
- **`reuseSession` and `includeHistory` on scheduled jobs** — two independent options control session behavior across runs. `reuseSession` accumulates all runs in one session instead of creating new ones each time. `includeHistory` (only when reuseSession is on) controls whether the agent sees previous run context. Four combinations cover independent runs, monitoring logs, and continuity patterns.

### Fixed

- **`executeChildSession` resumeFromHistory loaded empty parts** — the `resumeFromHistory` path in `executeChildSession` (`packages/server/src/core/child-session.ts`) used `storeListMessages()` which returns messages without parts, then mapped them with `parts: []`. The LLM received zero context from prior turns. This silently broke `task` tool session resume (via `task_id`) and scheduler reuse sessions. Fixed by switching to `buildEffectiveContextHistory()` which loads full `MessageWithParts[]` with all parts (text, tool calls, tool results, images), the same function `handleChat` uses.
- **Editing the first message of a session** — `revertToStep` treated `targetIndex === 0` (first message) as a "clear the entire session" operation, deleting the target message itself. This was correct for the revert feature but broke the edit flow: after editing the first message, the edited message was deleted, then the regeneration step tried to reuse a non-existent message, producing an empty-context reply. Added a `keepTarget` option to `revertToStep`; the edit handler (`handleSessionEditMessage`) now passes `keepTarget: true` so the target message is always preserved. The revert handler is unaffected (keeps the clear-all behavior for the first message).
